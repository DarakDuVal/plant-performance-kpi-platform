# Functional requirements

What the **plant performance KPI platform** must do. Each functional requirement describes one behaviour, is traceable
to the [user story](user-stories.md) that motivates it, and carries a Given-When-Then acceptance criterion a test can
assert. When a slice is ready to build, baseline it into an initiative spec from
[`../product/requirements-template.md`](../product/requirements-template.md).

**How to use:** give each requirement a stable `FR-NNN` id (never reused). Keep it atomic — split "the system shall do X
and Y" into two ids. Prioritise (`MUST` / `SHOULD` / `COULD`) and set a target iteration so requirements are resolved
stepwise. Track the **Spec status** so drift stays visible.

## Open questions & dependencies

- **CF (current forecast) cycles:** `CF02`, `CF05`, `CF09`, `CF11` denote the current-forecast cycles produced in
  February, May, September, and November respectively. Recorded here for traceability; move to the project context
  artifact once it exists.
- **Entities:** `ME` = Mobility Electronics (the business-unit total). Production plants (current list, subject to
  change): AdP, AnP, BhP, BrgP, CeaP, CljP, HtvP, JuP1, NhP1, PgP1, RBAC, RBEM, RtP2, SzP. Recorded here for
  traceability; move to the project context artifact once it exists.
- **SMT:** Surface Mount Technology — an example production process a plant may not have, triggering the FR-022
  not-applicable exception.
- **Maximum achievable score (FR-026):** Defined as the sum of the achievable scores of the KPIs that count for a
  plant — KPIs with an achievable score of 0 are excluded; not-applicable KPIs are still counted since they grant full
  points.
- **Roles:** *Regular user* = read-only viewer (FR-030). *Admin user* = authenticated user who may modify / correct
  data interactively (FR-029, FR-031). *Automation client* = authorized non-interactive programmatic data provider
  (FR-032). *Plant controller* = privileged user who may add contextual notes to a KPI (FR-041). *Super user* =
  privileged user who may edit the per-cell availability notes (FR-061, FR-062). Data enters the system by being pushed
  into the API (FR-028). A user's role is derived from their oneIDM / Entra ID group membership supplied in request
  headers (FR-033).
- **Reporting frequency:** KPIs are reported either monthly or quarterly (FR-045). Some are currently only quarterly
  — for example accident rate and sickness quota.
- **Gauge KPIs:** Some KPIs have no monthly/quarterly trend and are shown as a gauge versus target (FR-046) — e.g. CUV
  (Capacity Utilization Variance) compensation rate in %, and VA (Value-add) target gap in million EUR.
- **Cost-relevant KPIs:** KPIs flagged for the cost (Pareto) view (FR-049/FR-050) — e.g. TNS (Total Net Sales) and
  personnel cost.
- **Availability states (FR-055):** Each KPI/plant combination is one of *true* (green), *false* (red), *limited*
  (yellow), or *invalid* (grey). A plant that lacks the relevant production (e.g. no SMT) cannot achieve the related
  KPIs — the same situation that triggers the FR-022 not-applicable scoring exception.
- **Availability drives scoring:** The availability state (FR-055) governs ranking eligibility — *invalid* grants the
  full achievable score (FR-022), *false* and *limited* score 0 (FR-056), and *true* is scored normally (FR-019 /
  FR-020 / FR-021).
- **KPI notes on quarterly KPIs (FR-041) — to confirm:** Notes attach to a KPI + month; confirm how a note attaches for
  a quarterly KPI (e.g. to the quarter, or to a representative month).

**Spec status key:** `Specified` (not built) · `Implemented` (built) · `Verified` (built and covered by a passing,
traceable test) · `Drifted` (code and spec disagree — reconcile before the iteration closes).
